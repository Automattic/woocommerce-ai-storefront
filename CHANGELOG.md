# Changelog

## [Unreleased]

### Performance

- **`get_stats()` now caches its result in a transient for 5 minutes, keyed by period.** Pre-fix, every admin settings page load ran two `$wpdb` queries (one joining across orders and meta tables, one counting all-store orders) regardless of how recently the data had been computed. On stores with large order volumes the joins were visibly slow. Post-fix, the first call populates a `wc_ai_storefront_stats_{period}` transient; subsequent calls within the 5-minute window read directly from cache with zero DB queries. The cache is busted automatically on `woocommerce_order_status_completed` and `woocommerce_order_status_processing` events, so the dashboard reflects new order completions within one TTL cycle (at most 5 minutes). All four period transients are also deleted on plugin uninstall, including in multisite network uninstalls. Closes #160.
- **`get_catalog_summary()` result is now cached in a transient for one hour.** Pre-fix, every homepage and shop page request issued a fresh `get_terms()` database query regardless of whether the category list had changed. On stores with many categories this was a recurring synchronous DB hit on the critical path. The result is now stored under the `wc_ai_storefront_catalog_summary` transient key (TTL: `HOUR_IN_SECONDS`) and invalidated by `WC_AI_Storefront_Cache_Invalidator::invalidate()` on any product, category, or settings change. Closes #167.
- **`enhance_product_data()` no longer issues N+1 DB queries for category ancestors.** Pre-fix, `get_ancestors()` and `get_term()` were called once per category, and `get_ancestors()` in turn called `get_term()` for each level in the hierarchy. On a product with 3 categories each 3 levels deep this produced 12+ DB queries per product. A call to `_prime_term_caches()` before the loop loads all category and ancestor term objects in a single query so every subsequent `get_term()` call inside the loop is a cache hit. Closes #166.
- **`get_weight_unit_code()` and `get_dimension_unit_code()` now cache their result per instance.** Pre-fix, both methods called `get_option()` on every invocation. These methods are called once per product in the JSON-LD output loop; on a shop archive with 20 products this was 40 redundant `get_option()` reads per page. Both methods now cache the resolved unit code in a private instance property so `get_option()` is called at most once per `WC_AI_Storefront_JsonLd` instance (i.e. once per request). Closes #170.

### Security

- **Rate-limit fingerprint changed from raw UA string to `md5(bot_name + '_' + IP)`.** Three related weaknesses in the previous scheme are closed: (1) UA spoofing â€” any IP could claim another bot's user-agent and consume that bot's rate-limit budget without the real bot being aware; (2) UA rotation â€” a single IP sending minor UA variants of the same bot (e.g. `GPTBot/1.0` vs `GPTBot/2.0`) got a fresh 60-second window for each variant because `md5($full_ua)` produced a different key each time; (3) unknown agents with `allow_unknown_ucp_agents=yes` were never counted against any budget since the old code returned `true` immediately for unrecognized UAs â€” that path now issues a `md5('unknown_' + $ip)` fingerprint so those requests are rate-limited by IP like known bots. `REMOTE_ADDR` is used (not client-supplied forwarding headers) because forwarding headers are trivially forgeable and would allow unbounded unique IPs to bypass per-IP windows. `fingerprint_ai_bots()` (the WC Store API filter) is updated with the same bot-name+IP key. Closes FIND-S01.

### Fixed

- **`parse_ucp_id_to_wc_int()` now rejects malformed IDs with trailing non-digit characters.** Pre-fix, the `(int)` cast silently truncated `prod_123abc` to `123`, potentially resolving a spoofed or corrupted ID to a real product. A `ctype_digit()` guard now rejects any suffix that contains non-decimal characters, returning 0 (not-found) instead. The known `_default` suffix on synthesized default-variant IDs (e.g. `var_123_default`) is stripped before the check so legitimate round-trips continue to work. Closes #154.
- **`fetchStats` and `fetchEndpoints` now dispatch `setStatsError` / `setEndpointsError` on failure instead of silently swallowing errors.** Pre-fix, both async thunks had empty `catch` blocks, so any network or server failure left the Redux store with no observable error signal. Consumers had no way to distinguish "not yet loaded" from "failed to load." The fix adds `SET_STATS_ERROR` and `SET_ENDPOINTS_ERROR` action types, corresponding plain action creators, new `statsError` and `endpointsError` state slices in the reducer, and `getStatsError` / `getEndpointsError` selectors following the same pattern used for the rest of the store. A successful subsequent fetch clears the error so the UI can transition cleanly from failure back to data. Closes #161.
- **Warm-up cache write now targets the host-keyed transient (`host_cache_key()`) instead of the base `CACHE_KEY` constant.** Pre-fix, `update_settings()` wrote the freshly-generated `llms.txt` to `CACHE_KEY` on plugin enable, while `serve()` reads from `host_cache_key()` â€” the two keys are different on any real install (the host key is `CACHE_KEY . '_' . md5(HTTP_HOST)`), so the eager warm-up always caused a cache miss on the next request. Fix aligns the write site with the read site. Closes #151.
- **`get_recent_orders` now guards against `wc_get_orders()` returning `WP_Error` on DB failure.** Pre-fix, iterating a `WP_Error` object caused a `TypeError: argument of type WP_Error is not iterable` fatal in PHP 8, crashing the admin panel. With the guard, the endpoint returns HTTP 200 with an empty `orders` array so the UI shows "no orders" instead of a stack trace. Closes #155.
- **Four identical `array_map('absint', ...)` closures in the `product-count` route args are replaced by a single `sanitize_id_array` public static helper.** No behavior change; the method reference form (`array( __CLASS__, 'sanitize_id_array' )`) is functionally equivalent to the closures and avoids four copies of the same inline logic. Closes #162.
- **`search_categories()` and `fetch_flat_taxonomy_terms()` now pass `'number' => 500` to `get_terms()`.** Pre-fix, no limit was set, so a store with 10,000+ tags could return a 10 MB payload on every settings-page load. The cap of 500 covers all realistic stores; merchants who need more can use the search-as-you-type path that already has its own pagination. Closes #169.
- **Legacy `product_selection_mode` values (`categories`, `tags`, `brands`) are now normalized to `by_taxonomy` at write time in `update_settings()`.** Pre-fix, writing a legacy value through the REST settings endpoint or a direct `update_settings()` call stored the raw legacy string in the DB. The `get_settings()` silent migration then rewrote it on the next read, but a concurrent PHP worker reading between the write and the migration could observe the un-migrated value. Normalizing at write time keeps the DB clean from the moment of save. Closes #156.
- **`class_exists` guard removed from the Logger call in `get_settings()`'s migration error path in `class-wc-ai-storefront.php`.** The guard was inconsistent with every other Logger call site in the codebase; `WC_AI_Storefront_Logger` is always loaded as part of `load_dependencies()` during plugin bootstrap, so the guard was never false in production. Removed to match the direct-call pattern used throughout the other files. Closes #163.
- **`update_settings()` now resets `$settings_cache` before calling `get_settings()` to read the current persisted value.** Pre-fix, the `wp_cache_delete()` call that preceded `get_settings()` had no effect because `get_settings()` short-circuits on the non-null static cache before ever consulting the WP object cache. The fix nulls `$settings_cache` first so the merge in `update_settings()` always starts from the current DB value rather than a stale in-process snapshot. Closes #164.
- **`prime_syndication_cache()` static helper added to batch-prime term relationship caches before bulk `is_product_syndicated()` loops.** In `by_taxonomy` mode, each `is_product_syndicated()` call issues 1-2 DB queries via `wp_get_post_terms()` unless the term relationship cache is primed. Callers in the UCP REST controller can now call `prime_syndication_cache( $product_ids, $settings )` once before iterating to batch-load all term relationships in a fixed number of queries. The `is_product_syndicated()` docblock documents the batch-priming contract. Infrastructure for #168; the call site in the UCP REST controller is tracked separately.
- **`normalize_store_api_data()` now uses a recursive cast instead of a JSON round-trip.** Pre-fix, the method encoded the Store API response to JSON then decoded it, allocating a large intermediate string for no semantic benefit. Replaced with an O(N) recursive closure that casts `stdClass` objects to arrays in place. Output is identical; allocations are lower on large product payloads. Closes #171.
- **Redundant `class_exists('WC_AI_Storefront_Logger')` guards removed from `class-wc-ai-storefront-llms-txt.php` and `class-wc-ai-storefront-ucp-store-api-filter.php`.** The Logger class is always loaded as part of the plugin bootstrap via `load_dependencies()`, so the guard was unnecessary. Direct calls replace the wrapped blocks, matching the pattern used throughout the rest of the codebase. Partial fix for #163 (the remaining occurrence in `class-wc-ai-storefront.php` is addressed in a sibling PR).
- **`$_SERVER['REQUEST_METHOD']` reads in `class-wc-ai-storefront-llms-txt.php` and `class-wc-ai-storefront-ucp.php` now wrap the value in `wp_unslash()`.** The unslash call is a no-op for `REQUEST_METHOD` (server method names never carry magic-quote escaping), but it satisfies the WordPress coding-standards rule requiring `wp_unslash()` on all superglobal reads and is consistent with every other `$_SERVER` read in the codebase. The `class-wc-ai-storefront-ucp.php` call also gains an `isset()` guard for consistency with the llms-txt counterpart. Closes #165.
- **`uninstall.php` now purges all host-keyed `llms.txt` transient variants on plugin deletion.** Pre-fix, only the bare `wc_ai_storefront_llms_txt` transient was removed. Since 0.6.6 the actual cache uses host-keyed transients (`wc_ai_storefront_llms_txt_<md5(HTTP_HOST)>`); those rows were never deleted on uninstall, leaving orphaned option rows in the database. Fixed with a `$wpdb->query()` wildcard delete (`LIKE '_transient_wc_ai_storefront_llms_txt_%'`) in both the single-site block and the multisite loop. The same fix is applied to `deactivate()` in the cache invalidator so lingering rows are also swept on plugin deactivation. Closes #152.
- **Cache invalidation now purges all host-keyed `llms.txt` transient variants, not just the current-request host.** Pre-fix, `WC_AI_Storefront_Cache_Invalidator::invalidate()` called `delete_transient( host_cache_key() )`, which only removed the cache entry for the `HTTP_HOST` active at the time the invalidating request fired (typically a wp-admin request on the primary domain). A site reachable via both `example.com` and `www.example.com` would retain a stale cache on the alias domain for up to one hour after a product update. Fixed with the same wildcard `$wpdb->query()` pattern used in `uninstall.php`. Sites using a persistent object cache (Redis/Memcached) will still expire the alias-domain entry at its natural `HOUR_IN_SECONDS` TTL -- that is a documented limitation, not a regression. Closes #153.
- **`get_catalog_summary()` now purged by `invalidate()`.** The catalog summary transient (`wc_ai_storefront_catalog_summary`, introduced in this release for fix #167 below) is deleted on every product/category/settings change so the store-level JSON-LD reflects the latest data on the next homepage or shop page request.

## [0.6.6] â€“ 2026-04-28

### Fixes
- **`/llms.txt` and UCP manifest responses now emit `Vary: Host`; the llms.txt PHP-layer cache is keyed by `md5(HTTP_HOST)` rather than a bare constant.** Pre-fix, a reverse proxy or CDN keyed only on the request URL could serve a cached `/llms.txt` body from host A to a subsequent request from host B â€” injecting endpoint URLs that point at a foreign virtual host into the AI crawler's grounding document. The two-layer fix: (1) `Vary: Host` instructs every HTTP cache to maintain separate entries per `Host` header value; (2) the PHP-layer transient for llms.txt is now segmented by host using `CACHE_KEY . '_' . md5(HTTP_HOST)` so two concurrent WordPress instances sharing the same object-cache (e.g., a WP multisite with a Redis backend) cannot serve each other's bodies through the PHP layer. The UCP manifest computation is cheap (no HTTP probes), so its transient is dropped entirely in favor of per-request generation, removing the PHP-layer race without a keying change. Existing stale transients under the old bare key are harmless â€” they expire at their natural 1-hour TTL and are never read by the new code paths. Closes #125.
- **JSON-LD output now hex-escapes `<`, `>`, `&`, `'`, `"` to prevent script-tag breakout via taxonomy / site name strings.** Pre-fix, `wp_json_encode( $data, JSON_UNESCAPED_SLASHES | JSON_UNESCAPED_UNICODE )` let `</script>` survive literal in the homepage/shop JSON-LD body. A category name like `</script><script>alert(1)</script>` (creatable by any user with `manage_categories`, typically Editor role) escaped the script-tag CDATA context â€” stored XSS reachable on every page emitting JSON-LD. Replaced with `JSON_HEX_TAG | JSON_HEX_AMP | JSON_HEX_APOS | JSON_HEX_QUOT | JSON_UNESCAPED_UNICODE` across the three encode sites: `output_store_jsonld()` (the actual XSS vector), `WC_AI_Storefront_Ucp::serve_manifest()` and `WC_AI_Storefront_Admin_Controller::update_settings()` (defense-in-depth â€” manifest cache served as application/json so HTML breakout doesn't apply, but uniform encoding eliminates any future re-emission risk). Schema.org parsers and Google's structured-data validator handle hex-escaped characters correctly per the JSON spec; international (non-ASCII) text is unaffected because `JSON_UNESCAPED_UNICODE` is retained. Closes #122.
- **`/checkout-sessions` now collapses duplicate line items targeting the same product.** Pre-fix, an agent posting two entries for `prod_42` (one at qty 1 and one at qty 2) got back a response echoing both lines with totals summing to qty 3, but the buyer's cart at `/checkout-link/` contained qty 2 â€” WC's `/checkout-link/` parser uses each `id` as a key in its add-to-cart loop, so the second occurrence overwrote the first. Buyer paid different total than agent reported. Now `$processed` is collapsed by `wc_id` before `build_continue_url` and the response echo run; quantities sum, response shows one line per product, `continue_url` carries the summed quantity, and a new `merged_duplicate_items` info-message surfaces the collapse so agents can reconcile their sent payload against the response. Summed quantities exceeding `MAX_QUANTITY_PER_LINE_ITEM` are dropped with an `invalid_quantity` error whose `content` includes the offending `ucp_id` and the summed quantity (matching `process_line_item`'s single-line over-cap behavior). Closes #123.
- **Attribution strict gate no longer fires false-positive when `utm_id=woo_ucp` is only present in `$_GET` at order-creation time.** Pre-fix, `capture_ai_attribution()` checked both the order's stored `_wc_order_attribution_utm_*` meta AND the live `$_GET['utm_id']` / `$_GET['utm_medium']` values in the STRICT gate. A non-AI customer whose checkout URL happened to carry `utm_id=woo_ucp` (e.g. a forwarded ChatGPT link pasted into a different browser tab) got the order stamped as "Other AI", inflating the AI-channel stats with false positives. Fix: STRICT now reads only the stored order meta, which WC core wrote from the actual landing-page session. `$_GET` at order-creation time is a different navigation context and not a reliable attribution signal. The LENIENT gate's `$_GET['utm_source']` fallback (hostname matching against `KNOWN_AGENT_HOSTS`) is retained because it covers agents that build their own checkout URLs without going through our `/checkout-sessions` endpoint and may not trigger WC core's session-layer attribution capture. Closes #126.
- **UCP rate limiter now counts one slot per outer request, not one per inner Store API call.** Pre-fix, the `woocommerce_store_api_rate_limit_options` filter fired on every inner `rest_do_request()` dispatch that UCP route handlers issue internally â€” meaning a single `/catalog/lookup` with 50 product IDs consumed 50 rate-limit slots, leaving the agent with 0-5 slots remaining in the 60-second window for follow-up calls and triggering unexpected 429s on legitimate traffic. Fix: (1) `configure_rate_limits()` now returns `enabled: false` when `WC_AI_Storefront_UCP_Store_API_Filter::is_in_ucp_dispatch()` is true, suppressing WC's built-in per-call counter for inner dispatches; (2) a new `check_outer_rate_limit()` static method enforces a transient-based per-fingerprint (per-AI-bot-UA) counter at the outer-request layer, called from `check_agent_access()` on all allowed-through paths. The net result: 1 outer UCP request = 1 slot consumed, regardless of how many inner Store API calls it fans out to. Direct `/wc/store/v1/...` requests from AI bot UAs (bypassing UCP routes) remain rate-limited per WC's existing behavior since those requests run outside the UCP dispatch bracket. The merchant's `rate_limit_rpm` setting now reflects outer-request semantics. Closes #128, #129.
- **Per-product final-sale JSON-LD override no longer silently disappears when the WC base country is unset.** Pre-fix, the entire `hasMerchantReturnPolicy` block (including the per-product override that emits `MerchantReturnNotPermitted`) was nested inside a `if ($country &&...)` guard shared with `shippingDetails`. Installs that skip the setup wizard or run headless storefronts with no store address set got neither block. Fix: moves the return-policy emission out of the country gate into its own independent block. `shippingDetails` (which requires a country to form a valid `DefinedRegion`) remains gated. For `MerchantReturnNotPermitted` paths (per-product flag and store-wide final-sale mode), `applicableCountry` is emitted when the country is known and omitted when it is not â€” Schema.org marks the field as recommended, not required, for no-return policies. `returns_accepted` mode still requires a country (a return window without a target region is not useful). Closes #124.
- **Attribution no longer double-writes `_wc_ai_storefront_agent_host_raw` when both the lenient gate and the `ai_agent_host_raw` URL param fire on the same request.** Pre-fix, `capture_ai_attribution()` wrote the KNOWN_AGENT_HOSTS-normalized hostname to the meta (lenient-path branch, line ~499) and then unconditionally overwrote it with the URL-param value (lines ~546-548) when `?ai_agent_host_raw=...` was also present. The URL param flows through the buyer's browser so it is untrusted; a tampered param carrying a hostname-shaped value (e.g. `evil.example.com`) could silently displace the KNOWN_AGENT_HOSTS-validated value in the admin drill-in display. Fix gates the URL-param write on `!$is_known_ai_host`: when the lenient gate already wrote the normalized hostname, the URL-param path is suppressed entirely. The normalized KNOWN_AGENT_HOSTS value is always authoritative when available. Closes #127.
- **`allow_unknown_ucp_agents` setting now round-trips correctly through the REST settings endpoint.** The admin controller's `$fields` allow-list was missing the field, causing saves from the Settings UI to silently discard the value â€” merchants who toggled the control saw it snap back on next page load. The field was always honoured at read time; only the save path was broken. Closes #142.
- **JSON-LD return policy `returnFees` and `returnMethod` are now validated against Schema.org allow-lists at emit time.** Save-time sanitization in the admin controller is the primary gate; the emission-time check is a defense-in-depth layer guarding against DB values written by direct SQL, imports, or third-party plugins that bypass the sanitizer. An unknown `returnFees` string defaults to `FreeReturn`; unknown entries in `returnMethod` are silently filtered out; duplicate methods are deduplicated via `array_unique()`, mirroring save-time behavior exactly. Closes #141.
- **`wc_ai_storefront_jsonld_product` and `wc_ai_storefront_jsonld_store` filters now receive a minimal settings subset instead of the full settings array.** Pre-fix, the full array (including `rate_limit_rpm`, `allowed_crawlers`, `allow_unknown_ucp_agents`, and per-product selection arrays) was passed as the third argument to both filters. Any third-party filter callback reading those fields could observe internal configuration not intended for external consumers. The argument is now a 3-key subset: `enabled`, `product_selection_mode`, `return_policy`. Filter callbacks that previously read the removed keys will silently receive `null` â€” a breaking change for any code that depended on those keys, but the documented filter signature never described them as public API. Closes #144.
- **`meta.source` in `/checkout-sessions` POST body is now validated before use in attribution.** Pre-fix, the body-fallback agent-identification path accepted any string as `meta.source` and stored it in order meta and the raw-host attribution field without format checks. The value is now validated against the same token charset as the UCP-Agent Product/Version header path: letters, digits, hyphens, dots, and underscores; max 253 chars. Values failing validation fall through to the empty-agent sentinel, matching the behavior of the other identification paths. Closes #143.
- **llms.txt sitemap probe now enforces SSL certificate verification in production.** The `wp_remote_head()` call that probes the site's XML sitemap URL unconditionally passed `'sslverify' => false`, disabling certificate validation in all environments. Fixed to `! ( defined('WP_DEBUG') && WP_DEBUG )` â€” verification is enabled in production and disabled only in local development environments where self-signed certificates are common. Closes #145.

### Tests
- **Two test quality issues addressed in the JSON-LD and attribution suites.** Fix 1 (`add_query_arg` mock fragility, #138): the shared `add_query_arg` mock in `JsonLdTest`, `JsonLdNormalizationTest`, and `JsonLdReturnPolicyTest` did not strip URL fragments before appending query params â€” a permalink like `https://example.com/product/widget/#tab-description` produced `...#tab-description?add-to-cart=42`, with the query string absorbed into the fragment and invisible to the server. Mock updated to match core's behavior; regression test added. Fix 2 (STRICT-gate sentinel leak, #139): three `AttributionTest` tests used real agent hostnames (`chatgpt`, `gemini`, `claude`) as `utm_source` values in positions where agent identity is irrelevant to the assertion. If any name were ever added to `KNOWN_AGENT_HOSTS`, the lenient gate would silently also fire, changing the test's semantics without any warning. Replaced with `mysteryagent.example` â€” an RFC 2606 reserved domain guaranteed never to be a real `KNOWN_AGENT_HOSTS` entry. Closes #138, #139.

### Docs
- **`CART-MODELS.md` added to engineering documentation.** Documents all four cart/checkout architectures the plugin supports: Model 1 (surface-only links â€” agent links bare product URLs); Model 2 (agent calls `/checkout-sessions` with full `items[]`, receives `continue_url`); Model 3 (agent constructs WC Shareable Checkout URLs directly, zero per-change network cost); Model 4 (stateful `dev.ucp.cart` capability with `/cart-sessions` endpoints on the roadmap, documented trigger conditions). All four models preserve merchant ownership of checkout. Cross-linked from `AGENTS.md`, `docs/README.md`, and `docs-followup.yml`. Addressed by #147.

## [0.3.2] â€“ 2026-04-27

### Features
- **Stub `GET / PUT / PATCH / DELETE /checkout-sessions/{id}` returns a structured 405 `unsupported_operation` response.** Background: UCP agents that come from a stateful-session mental model try to act on a session via several verbs â€” PATCH to modify the cart, PUT to replace the payload, GET to look up status, DELETE to cancel. Our `/checkout-sessions` model is intentionally stateless: every POST is a fresh computation returning a one-shot Shareable Checkout redirect, no persisted session for any verb to act on.
